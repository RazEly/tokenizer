## Training

Throughout training I set many hyperparameters to optimize differently for each domain. I then performed a grid search and measure F1, speed and efficiency and eventually hard-code the best performing parameters for each domain.

**Pre-tokenization:** I implemented regex a pre-tokenization step, using the base regex from the open-source GPT-2 tokenizer.

**Length-weighted merge score**: the merge priority is $\text{score}(a, b) = \text{freq} · (len(a)+len(b))^\alpha$ rather than frequency alone. As $alpha$ increases the merging prioritizes already-long symbols. This is done to improve the tokens-per-character metric. $\alpha$ is tuned per domain.

**Two-stage merging:** Stage 1 is a classic BPE, stopping at the word level. In stage B we observe only adjacent words that are already compressed to one token and merge them.
We use hyperparameters to control the number of merges in each stage, and the minimum frequency of pairs to merge. We also ensure that at least one bigram is always learned, to adhere to the requirements of the assignment.

**Heap + inverted index:** I used a heap to reduce runtime complexity. Selecting the next merge is a heap pop. I then use the `pairs_to_words` inverted index to find the words that contain the pair, and re-merge only those words, instead of iterating over the entire corpus.

- `pair_freqs`: the current count of every adjacent pair, summed with word frequency weight.
- `pair_to_words`: an inverted index mapping each pair to the set of word indices that contain it.
- a binary max-heap ranked by merge score.

- Selecting the best pair is a heap pop.
- I use **lazy deletion**: every popped entry is re-scored against the live `pair_freqs`, and if the stored score disagrees it is stale and discarded
- Fresh entries for changed pairs are simply pushed on top
- the stale copies sink and are skipped when they surface.

- After choosing `best`, only the words in `pair_to_words[best]` can possibly change, so I re-merge just those words, diff their old vs. new adjacent-pair multisets (`Counter(zip(...))`), and apply the deltas to `pair_freqs` and the index.
- The cost of a merge is therefore proportional to the number of affected words instead of the whole corpus.

**Heap + array doubly-linked-list encoder (`_bpe`, encoding):**

- encoding a chunk is a different problem.
- what I need is to apply the learned merges to one short symbol sequence *in rank order* — earliest-learned merge first, exactly reproducing training-order BPE.
- An **array-backed doubly-linked list** (`prev`, `nxt`, `alive`) over the symbol positions. Merging a pair means splicing one node out — flip `alive[j]=False`, relink `nxt[i]`/`prev[...]` — in $O(1)$, with no list shifting. Using parallel integer arrays instead of node objects keeps it allocation-light and cache-friendly.
- A **min-heap keyed by merge rank** (`bpe_ranks[pair]`, lower = learned earlier). Popping always yields the globally lowest-rank applicable pair, so I never scan.

The matching lazy-deletion trick appears here too: when a merge consumes a node, its neighbours' pairs change, so I push the *new* neighbour pairs and let the old heap entries go stale. A popped entry is validated three ways before use — both endpoints still `alive`, still adjacent (`nxt[i] == j`), and the live `(syms[i], syms[j])` still has the rank that was pushed — and skipped otherwise. This brings per-chunk encoding to roughly $O(n \log n)$. Results are cached per chunk string, so the cost is paid once per distinct pre-token.

!!!**Two-level caching:** `self.cache` memoizes chunk $\to$ symbols and `self._encode_cache` memoizes full text $\to$ ids (with lazy init for tokenizers pickled before the cache existed).

## Evaluation — tokenizer 3 on the hidden domain

F1 on the hidden domain is not directly measurable, so we reason about transfer. Strengths: (a) full byte-level base → zero OOV on any input; (b) merges learned from ~1.6M mixed lines cover casual *and* formal registers; (c) minimal bigram reliance avoids overfitting source-domain phrases; (d) handle stripping removes Twitter-specific noise. Risk: a highly specialized hidden domain (biomedical, code) would segment into longer subword sequences (worse efficiency) though still correct. We expect efficiency between the two known domains (~0.29 tokens/char) and F1 nearer the formal tokenizer than the social one, assuming the hidden text is cleaner than Twitter.
